# UI / UX デザイン方針

Local 版（`apps/v1`）の画面設計・インタラクション設計の考え方。  
ゲーム UI の **わかりやすさ・手応え・進捗感** を業務向けに取り入れるが、**ゲーム風の見た目にはしない**。

## 目的

ユーザーが自然に理解できるようにする。

- 今どこを見るべきか
- 何が重要か
- 次に何をすべきか
- 操作の結果、何が起きたか
- 作業は前に進んでいるか
- 問題がある場合、どこを直せばよいか

BtoB としての **信頼感**（静か・正確・邪魔しない）を損なわない。

## 基本スタンス

| 取り入れる | 取り入れない |
|---|---|
| 注意の誘導 | 派手な色・ネオン・グラデーション |
| 操作への即時フィードバック | 装飾的フォント・キャラクター性 |
| 状態変化の可視化 | 長い演出・常時ループアニメ |
| 進捗感・小さな完了感 | ポイント・ランキングの安易な導入 |
| 次アクションの明確化 | 毎回の祝福演出（🎉 等） |
| 不安を減らす短いモーション | 作業を止めるモーダルの多用 |

静的な管理画面ではなく、**判断と行動を導く操作空間** を目指す。

## HUD 的な画面整理

テスト実行画面（`/run`）はダッシュボードではなく **作業 HUD** として整理する。

| 位置 | 要素 | 役割 |
|---|---|---|
| 上部 sticky | `FilterBar` | スコープ・件数・「未実施のみ」 |
| 左（desktop） | `RunnerTaskList` | 一覧・進捗・ジャンプ |
| 中央 | `TestCard` | **主操作**（1件ずつ集中入力） |
| 下部 fixed | `RunProgressBar` | フィルタ行 + 全体の進捗・残り件数 |
| 右下 | `ShortcutHelp` | キーボード一覧 |

マトリクス表は補助。ランナーカードが主役（[requirements.md](./requirements.md) の Local 版 UX と一致）。

プロジェクト一覧（`/projects`）も同様に **左一覧 + 右パネル** の HUD 構成とする。

| 位置 | 要素 | 役割 |
|---|---|---|
| 上部 | ページヘッダ | タイトル・説明 |
| 左（desktop） | `ProjectList` | 新規作成 + 登録済みプロジェクト一覧 |
| 右 | `ProjectImportPanel` / `ProjectDetailPanel` | 選択に応じた import または管理操作 |

## 重要度ごとの強調（Level 1〜5）

同時に Level 3 以上を出す要素は **1画面あたり 1〜2 個** に抑える。

| Level | 用途 | 表現例 |
|---|---|---|
| 1 通常 | 完了済み、SKIP、前提条件 | 静的表示、`text-muted-foreground` |
| 2 少し注意 | 残り件数、次へボタン、フィルタ件数 | バッジ、薄い背景、`text-primary` |
| 3 要対応 | 未入力端末、NG、バリデーション | 色・位置・枠（控えめ） |
| 4 緊急 | 読み込みエラー、必須未入力 | 短い shake + `Alert` |
| 5 致命的 | データ破損等（稀） | 操作ブロック + 明確な警告 |

**強調の原則**

- すべてを目立たせない。本当に注意すべきものだけ。
- アニメーションは **意味がある場面** に限定する。
- NG は行・バッジで示す。画面全体を赤くしない。

## アニメーション

装飾ではなく **「何が起きたか」** を伝えるために使う。

| 用途 | 目安 | 実装例 |
|---|---|---|
| hover / press | 100〜150ms | ボタン `active:scale-[0.97]` |
| 小さな状態変化 | 150〜250ms | `StatusBadge` の fade + zoom |
| 追加・削除・遷移 | 200〜300ms | カード `slide-in`、ファイル行 fade-in |
| 注意喚起 | 300ms 以内 | `animate-ui-shake` |

**入れる場面の例**

- ステータスが変わった（バッジ・行ハイライト）
- テストカードが前後に進んだ（`RunnerCardTransition`）
- フィルタ件数・完了数が変わった（数値の短い fade-in）
- スコープ完了（完了カードのチェックアイコン）
- エラー（shake）

**避けること**

- 常時動き続ける要素
- 同画面で多数の要素を同時アニメーション
- 意味のないローディング演出

`prefers-reduced-motion: reduce` ではアニメーションを無効化する（`global.css`）。

## 操作フィードバック

操作後に反応がないと、保存されていない・止まっていると不安になる。

| 操作 | フィードバック |
|---|---|
| ボタン押下 | 軽い押し込み（scale）、OK/NG は brightness |
| ステータス入力 | バッジ切替、端末行・一覧行の一瞬ハイライト |
| カード遷移 | 方向付きスライド（前進 / 後退） |
| 進捗更新 | バーセグメント幅 transition、残り件数表示 |
| ファイル追加 | リスト行 fade-in |
| バリデーション失敗 | フィールド shake + メッセージ |

**意図的に入れていないもの**

- 画面上部の「保存中 / 保存済み」インジケータ（作業の邪魔になるため不採用）
- IndexedDB への自動保存は継続。UI では黙って保存する

## テストカード（主操作面）

### 視線の中心

- **確認内容**（`description`）を大きく・太く。ID・分類は補助情報。
- 前提条件は Level 1（控えめ）。

### 未入力の端末

どの環境から先に入れるかは **ユーザー次第**。  
「次に入力すべき行」をシステムが決めない。

| 状態 | 表現 |
|---|---|
| 入力済み | 通常枠 `border-border bg-muted/30` |
| 未入力 | `border-primary/35 bg-card shadow-sm`（全未入力行を同じ） |

「未入力」ラベルや先頭行だけの特別強調は使わない。  
未対応行がまとめて目に入る程度の差に留める。

### キーボードと `all` / `any`

ショートカット `o` / `n` / `s` は **表示中の全端末に一括** で同じステータスを記録する。

そのため、キーボード主体のランナー体験では完了条件 **`all`**（全端末そろいで完了）と相性が良い。  
`any` だと「1端末だけで次へ進む」動きと一括入力の意味がずれる。

サンプル `tests.yml` では大分類 `テスト実行` の `categoryTargets.required` を `all` にしている。  
ケース単位・分類単位の `any` 検証用 TC（例: TC-008）は個別の `targetEnvironments` で定義する。

## 進捗・完了

### 進捗 HUD（`RunProgressBar`）

- フィルタスコープ行 + 全体行の二段バー
- 右側に `N / M` と **残り件数**（Level 2）
- ホバーでステータス別内訳（既存）

### 完了カード

- 派手な演出は避け、チェックアイコンの短い scale-in
- 次アクション（フィルタ変更、セッション設定へ）を明示
- 🎉・強いグラデーションは使わない

### タスク一覧

- アクティブ行: 左 primary ボーダー
- NG 行: 控えめな赤背景（Level 3）
- ステータス更新直後: 短いハイライト（OK=緑、NG=赤）

## 技術実装（Local 版）

| 項目 | 方針 |
|---|---|
| スタイル | Tailwind CSS v4 + shadcn/ui |
| アニメーション基盤 | `tw-animate-css` + 少数の custom keyframes |
| カスタム motion | `global.css`: `ui-shake`, `ui-highlight-*` |
| 状態トラッキング | `AppContext.lastUpdatedTestId`（一覧ハイライト用） |
| カード遷移 | `RunnerCardTransition` + `TestRunner` の direction |

新規モーションを足すときは **duration・意味・Level** をこの文書に照らして判断する。

## 避ける表現（チェックリスト）

- [ ] ゲームっぽい派手な見た目
- [ ] 過剰なグラデーション・ネオン
- [ ] 長いアニメーション（>300ms の演出）
- [ ] 常時 pulse / bounce
- [ ] 重要でない情報への過剰な強調
- [ ] 保存完了の常時表示トースト
- [ ] 未入力端末の「次はここ」強制（先頭行だけ目立つ等）
