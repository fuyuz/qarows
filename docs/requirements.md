# 要件定義

## 1. プロダクト概要

| 項目 | 内容 |
|---|---|
| 名称 | **qarows** |
| 種別 | QA シート特化 Web アプリ |
| 主な利用者 | QA 担当、開発者 |
| 技術スタック | React + Vite、Cloudflare Pages |
| UI 言語 | 日本語メイン（将来 i18n で英語対応） |

### 解決したい課題

スプレッドシートで QA シート（テストケース × 端末/環境のマトリクス）を運用する際の課題を改善する。

- 行・列が多くてスクロールが大変
- ステータス入力が面倒
- フィルタ・絞り込みがしづらい
- 自分のテスト対象と関係ない端末/環境も見えて作業しにくい

---

## 2. エディション構成

Local 版と Team 版は **別アプリとして独立デプロイ・独立利用** できる状態を維持する。

| | Local 版（`apps/v1`） | Team 版（`apps/v2`） |
|---|---|---|
| 提供形態 | 公式インスタンスを固定 URL で公開 | **各自が専用の closed 環境**にセルフデプロイ（公式共通インスタンスなし） |
| 利用者 | 誰でも URL にアクセスするだけ | デプロイした組織内のメンバー（Cloudflare Access で限定） |
| プロジェクト数 | 複数（IndexedDB にローカル保存） | 複数プロジェクト |
| サーバー | 不要（ブラウザ完結） | 必要（Workers + DO 等） |
| データ保存 | IndexedDB（ローカル） | サーバー側に永続化 |
| 協調 | `results.json` のファイルマージ | リアルタイム同期 |

詳細なデプロイ方針は [deployment.md](./deployment.md) を参照。構成図は [architecture.md](./architecture.md) を参照。

---

## 3. Local 版要件

### 3.1 基本方針

- サーバー不要。ブラウザだけで完結する。
- サービス側はデータを持たない。
- テスト定義はファイルアップロード、作業内容は IndexedDB に自動保存。
- 編集結果は JSON / YAML でエクスポート可能。

### 3.2 データファイル

| ファイル | 役割 | 形式 |
|---|---|---|
| テスト定義 | テストケース、端末/環境、プロジェクト設定 | **YAML**（`tests.yml`） |
| 実行結果 | ステータス、メモ、バグ | **JSON**（`results.json`） |

詳細スキーマは [data-format.md](./data-format.md) を参照。

### 3.3 テストケース

| フィールド | 説明 |
|---|---|
| ID | 一意識別子 |
| 大分類 / 小分類 | 機能・モジュールの階層 |
| テスト前提条件 | テスト開始前に必要な状態（例: ログイン済み） |
| 確認内容 | 検証する具体的な項目 |

**実行結果**（テストケース × 端末/環境ごと）:

| フィールド | 説明 |
|---|---|
| ステータス | OK / NG / SKIP |
| 実施日 / 実施者 | いつ、誰が検証したか |
| メモ | 自由テキスト（複数行） |

**ステータス優先度**（弱 → 強）: `OK < SKIP < NG`

端末/環境は `tests.yml` 内で列挙し、プロジェクトごとに固定リストとする。

### 3.4 バグ管理

`results.json` に含める。

| フィールド | 説明 |
|---|---|
| 関連テストケース ID | |
| 重要度 | |
| 担当者 | |
| ステータス | 未対応 / 対応中 / 修正済み 等 |
| タイトル | |
| 再現手順 | |
| 期待 / 実際の動作 | |

### 3.5 役割分担

| 役割 | 操作 |
|---|---|
| QA 担当 | テストケース作成・修正、実行、結果入力 |
| 開発者 | バグ対応、修正確認 |

### 3.6 UX

メイン操作は **1 テストずつ集中入力**。スプレッドシート的なマトリクス全体表示は補助とする。

| 機能 | 仕様 |
|---|---|
| 絞り込み | 作業開始時に端末/環境を手動選択。大分類・未実施のみ等のフィルタを都度設定 |
| 入力後 | 結果入力後、自動で次のテストへ |
| 戻る | 前のテストにすぐ戻れ、結果は上書き編集可能 |
| 一括入力 | 作業開始時に選んだ端末/環境すべてに一括 OK / NG |
| メモ | 自由テキスト 1 件（複数行表示） |

### 3.7 マージ（複数人のローカル作業）

各自が手元で作業した `results.json` をマージする。

| 項目 | ルール |
|---|---|
| マージ対象 | `results.json` のみ（`tests.yml` は対象外） |
| ステータス競合 | 強い方が勝つ（`OK < SKIP < NG`） |
| メモ競合 | 両方残す。改行で連結。`---` 等の区切り線を入れてもよい |

## 4. Team 版要件

### 4.1 基本方針

- **各自専用の closed 環境**。組織（または個人）ごとに Cloudflare アカウントへ独立デプロイし、他デプロイのデータやユーザーとは共有しない。
- 同一デプロイ内のメンバーだけがリアルタイム協調する（チーム内 QA）。
- 複数プロジェクトを管理する。
- テスト結果・メモ・バグ・テストケース定義を **すべてリアルタイム同期** する。
- **Cloudflare Access** でメールアドレス認証し、closed 環境への入口を限定する（Worker 側でも enforce）。
- 公式の共通 Team 版 URL は提供しない。セットアップは [deploy-v2.md](./deploy-v2.md) を参照。

### 4.2 プロジェクト作成

- Web UI 上で新規作成
- 既存 `tests.yml` をアップロードしてプロジェクト化

両方に対応する。

### 4.3 同時編集

**後から保存した人が上書き**（Last Write Wins）。

Local 版のマージルール（ステータス強度、メモ両方保持）とは別ルール。リアルタイム同期では LWW を採用する。

### 4.4 Local 版との関係

Team 版リリース後も Local 版（ファイル + IndexedDB + マージ）を残すかは **状況次第** で判断する。アーキテクチャ上は両方独立デプロイ可能な構成とする。

### 4.5 技術（想定）

| コンポーネント | 用途 |
|---|---|
| Cloudflare Pages | フロントエンド配信 |
| Pages Functions / Workers | API、WebSocket Upgrade 受付 |
| Durable Objects | 接続管理、プロジェクト単位の状態 |
| D1 等 | 永続ストレージ |
| Cloudflare Access | 社内メール認証 |
